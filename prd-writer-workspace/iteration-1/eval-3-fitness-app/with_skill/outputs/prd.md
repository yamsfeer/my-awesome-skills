## 1. 产品愿景

### 产品定位

一款面向健身爱好者的运动记录与打卡社交 APP，帮助用户坚持训练计划、量化进步轨迹，并通过朋友间的相互鼓励形成持续的运动习惯。

### 核心价值主张

- **记录即激励**：每次打卡都能直观看到数据积累，用进步曲线驱动用户坚持下去
- **计划先行**：支持创建个性化训练计划，不再漫无目的地健身
- **可视化成长**：将力量、耐力、体重等数据整合成清晰的进步图表，让努力有据可查
- **社交正能量**：朋友互相打卡鼓励，形成轻量级互动氛围，降低独自坚持的心理难度
- **低门槛上手**：不需要专业知识也能快速记录一次训练，5 分钟完成打卡

### 差异化优势

| 对比维度 | 本产品 | Keep | 健身兄弟（FitBro） | 手动记录（微信群/笔记） |
|---|---|---|---|---|
| 训练计划自定义 | 完全自定义 | 内置模板为主 | 支持 | 无结构 |
| 进步曲线可视化 | 核心功能 | 有但需付费 | 有 | 无 |
| 朋友打卡鼓励 | 轻量社交，聚焦鼓励 | 内容广场为主 | 弱 | 手动截图 |
| 学习成本 | 低，开箱即用 | 中等 | 中等 | 低但无沉淀 |
| 数据隐私 | 默认仅好友可见 | 公开为主 | 公开为主 | 私密 |

### 成功指标（上线后 6 个月内）

| 指标 | 目标值 |
|---|---|
| 注册用户数 | 50,000 |
| 日活用户（DAU） | 10,000 |
| 次日留存率 | ≥ 40% |
| 7 日留存率 | ≥ 25% |
| 打卡完成率（有计划的用户） | ≥ 60% |
| 平均每月打卡次数/活跃用户 | ≥ 8 次 |

## 2. 目标用户

### 核心用户画像

| 维度 | 描述 |
|---|---|
| 年龄 | 18 - 35 岁 |
| 性别 | 男女均有，男性略多（约 60%） |
| 地域 | 一二线城市为主 |
| 职业 | 上班族、学生 |
| 收入 | 中等及以上（有能力办健身卡） |
| 运动频率 | 每周 2 - 5 次，有一定健身基础但非专业运动员 |
| 行为特征 | 关注自身体型与健康，有计划运动的意愿，但容易因为缺乏记录和反馈而中断 |

### 典型使用场景

**场景 A：健身后记录**
> 小王，28岁程序员，每周去健身房3次。每次训练结束，他想记录今天做了哪些组、重量是多少，下次来能有参考。以前用备忘录记，零散混乱。他希望有个专门的地方，2分钟内快速完成记录，还能看到上次做同一动作的数据。

**场景 B：制定并执行计划**
> 小李，22岁大学生，刚开始健身，不知道该练什么。她想制定一个「5天增肌计划」，每天按计划打卡，完成后获得满足感。她希望计划完成进度一目了然，能感受到自己在认真执行。

**场景 C：朋友相互激励**
> 阿强和同事约好每周互相督促健身。他希望能看到对方今天打卡了没有，训练了什么，顺手给一个鼓励。不想要复杂的社区功能，只需要简单的「看到+鼓励」即可。

### 痛点分析

| 痛点 | 现有解决方式 | 不满之处 |
|---|---|---|
| 训练数据难以追踪 | 手写本、备忘录 | 无法查询历史、无图表、容易丢失 |
| 不知道自己有没有进步 | 自己感觉 | 主观，无数据支撑，容易放弃 |
| 容易三分钟热度 | 靠意志力 | 缺乏外部激励，无社交监督 |
| 朋友督促太麻烦 | 微信发截图 | 低效，没有专门的互动场景 |
| 训练计划不系统 | 看 YouTube/微信文章 | 无法与自己的记录数据联动 |

## 3. 功能需求

> 本章节按用户故事地图组织，覆盖用户使用产品的完整旅程。

### 3.1 用户活动总览

```
注册与设置 → 创建训练计划 → 日常打卡记录 → 查看进步数据 → 好友互动鼓励
```

---

### 活动一：注册与设置

#### 用户任务 1.1：注册账号

**用户目标：** 快速完成注册，开始使用，不被复杂流程拦住

**系统功能：**
- 支持手机号 + 验证码注册
- 支持微信授权一键登录
- 注册时填写昵称、头像（可跳过后补）
- 首次进入引导用户设置基本信息（身高、体重、健身目标）

**优先级：** P0

**验收标准：**
- 从打开 APP 到完成注册不超过 60 秒
- 微信登录成功率 ≥ 99%

---

#### 用户任务 1.2：设置个人信息

**用户目标：** 让 APP 了解我的基本状态，后续数据更有参考价值

**系统功能：**
- 可填写身高、体重、年龄、性别
- 可选择健身目标（增肌/减脂/提升耐力/保持健康）
- 信息可在「我的」页随时修改
- 体重支持历史记录（可手动录入每次称重结果）

**优先级：** P1

---

### 活动二：创建训练计划

#### 用户任务 2.1：新建训练计划

**用户目标：** 创建一个属于自己的、有结构的训练计划，知道每天该练什么

**系统功能：**
- 支持创建「周计划」（按周循环）或「阶段计划」（固定天数，如 30 天计划）
- 计划包含名称、描述、封面图（可选）
- 计划下可添加多个「训练日」，每个训练日可定义当天要完成的动作组合
- 训练日可设置为「休息日」

**优先级：** P0

**验收标准：**
- 用户可在 5 分钟内从零创建一个包含 3 个训练日的计划

---

#### 用户任务 2.2：编辑训练动作

**用户目标：** 在训练日中添加具体的动作，包括组数、次数或时长

**系统功能：**
- 支持从动作库搜索并添加动作（内置 100+ 常见健身动作）
- 支持手动创建自定义动作
- 每个动作可配置：组数、每组次数（或时长）、休息时间、重量（可选）
- 动作支持拖拽排序
- 动作详情页展示该动作的目标肌肉群和图示（图片）

**优先级：** P0

**验收标准：**
- 动作库搜索响应时间 < 500ms
- 内置动作库覆盖胸、背、腿、肩、手臂、核心六大肌群，各不少于 10 个动作

---

#### 用户任务 2.3：复制或参考系统模板计划

**用户目标：** 不想从零开始设计计划，希望直接使用或基于模板修改

**系统功能：**
- 系统内置 10+ 套常见训练计划模板（如：5天增肌分化、3天全身训练、新手入门等）
- 模板可一键复制为「我的计划」并自由修改
- 模板展示适合人群、训练目标、难度等级

**优先级：** P1

---

### 活动三：日常打卡记录

#### 用户任务 3.1：开始今日训练

**用户目标：** 按计划开始今天的训练，不需要重新查找要做什么

**系统功能：**
- 首页突出展示「今日训练」按钮，一键进入当天计划对应的训练日
- 若未设置计划，可选择「自由训练」（临时选动作，不依赖计划）
- 进入训练后展示当天所有动作列表，逐个完成

**优先级：** P0

---

#### 用户任务 3.2：记录每组训练数据

**用户目标：** 快速记录每组完成的重量和次数，不打断训练节奏

**系统功能：**
- 每组训练完成后，点击「完成」记录本组数据（次数/重量/时长）
- 自动带入上次同一动作的数据作为默认值，支持修改
- 组间计时器（休息倒计时），可自定义时长
- 支持为单次训练添加文字备注（如：感觉状态一般，睡眠不足）
- 支持添加训练照片（每次最多 3 张）

**优先级：** P0

**验收标准：**
- 完成一组记录（从点击完成到进入下一组）操作步骤不超过 3 步

---

#### 用户任务 3.3：完成打卡

**用户目标：** 训练结束后完成打卡，获得成就感，并让朋友看到

**系统功能：**
- 训练结束时展示「打卡完成」页面，包含：本次训练总时长、总容量（总重量×总次数）、完成动作数
- 可一键生成「打卡海报」（包含训练数据、日期、用户头像）
- 打卡记录默认可被好友查看，可设置为仅自己可见
- 支持将打卡海报分享到微信朋友圈/微信好友

**优先级：** P0

---

#### 用户任务 3.4：自由记录（非计划训练）

**用户目标：** 没有预设计划时，也能快速记录今天的临时训练

**系统功能：**
- 从动作库选择动作，自由组合成一次训练记录
- 记录完成后同样可以打卡、分享
- 自由训练记录纳入整体数据统计

**优先级：** P1

---

### 活动四：查看进步数据

#### 用户任务 4.1：查看训练历史

**用户目标：** 回顾过去每次训练的内容，找到规律

**系统功能：**
- 日历视图展示打卡记录（打卡日有标记），可点击查看当天详情
- 列表视图展示所有历史训练记录，按时间倒序
- 每条记录可展开查看完整动作组数和数据

**优先级：** P0

---

#### 用户任务 4.2：查看进步曲线

**用户目标：** 直观看到自己在某个动作上的重量/次数变化，感受到进步

**系统功能：**
- 按动作维度展示历史数据折线图（如：卧推最大重量变化趋势）
- 支持查看近 1 个月、3 个月、6 个月、全部的数据区间
- 汇总页展示整体运动统计：本月打卡次数、总训练时长、最长连续打卡天数
- 支持查看体重变化趋势（需手动录入体重）

**优先级：** P0

**验收标准：**
- 折线图数据点至少有 2 个时才展示图表，否则显示引导提示「继续记录，数据会在这里显示」

---

#### 用户任务 4.3：查看计划完成情况

**用户目标：** 了解当前训练计划的执行进度，知道哪里落下了

**系统功能：**
- 计划详情页展示各训练日的完成/未完成状态
- 展示计划整体完成率（如：已完成 18/30 天）
- 连续打卡天数和当前「打卡流」（streak）展示
- 若超过 3 天未按计划打卡，App 发送轻提醒通知

**优先级：** P1

---

### 活动五：好友互动鼓励

#### 用户任务 5.1：添加好友

**用户目标：** 找到认识的朋友，建立互相监督的关系

**系统功能：**
- 支持通过手机号搜索并添加好友
- 支持分享邀请链接给微信好友，对方点击可直接添加
- 好友列表展示对方昵称、头像、近期打卡天数

**优先级：** P0

---

#### 用户任务 5.2：查看好友动态

**用户目标：** 看到朋友最近有没有打卡，了解对方的训练情况

**系统功能：**
- 好友动态 Feed：按时间展示好友的打卡记录（不做算法推荐，仅时间顺序）
- 每条动态展示：头像、昵称、训练摘要（时长、动作数）、打卡时间、训练照片（若有）
- 不展示好友的具体组数/重量数据（保护隐私），仅展示摘要

**优先级：** P0

---

#### 用户任务 5.3：给好友打卡鼓励

**用户目标：** 看到好友打卡后，简单表示支持和鼓励

**系统功能：**
- 支持对好友打卡记录点「加油」（类似点赞，一条记录每人只能加油一次）
- 支持留言评论（文字，不超过 200 字）
- 发出的鼓励和收到的鼓励均有消息通知

**优先级：** P0

---

#### 用户任务 5.4：接收训练提醒与鼓励通知

**用户目标：** 在合适的时间收到提醒，不错过好友的鼓励

**系统功能：**
- 用户可设置每日训练提醒时间（如：每天 18:00 提醒）
- 收到「加油」或评论时推送通知
- 好友今日也打卡时，可推送「XXX 今天也打卡了，你也来吧！」（可关闭）

**优先级：** P1

## 4. 非功能需求

### 4.1 性能要求

| 指标 | 要求 |
|---|---|
| 首页加载时间（冷启动） | P95 < 2s |
| 动作库搜索响应时间 | P95 < 500ms |
| 好友动态 Feed 加载 | P95 < 1.5s |
| 打卡数据保存成功率 | ≥ 99.9%（本地先存，网络恢复后同步） |
| 服务可用性 | ≥ 99.5% |
| 并发用户支持 | 初期支持 1,000 并发，可横向扩展 |

### 4.2 安全要求

- 用户训练数据默认不对外公开，仅好友（双向关注）可见
- 支持设置单条打卡记录为「仅自己可见」
- 手机号等敏感信息不展示给其他用户
- 采用 HTTPS 传输，数据库中用户密码加盐哈希存储
- 第三方登录（微信）遵循 OAuth 2.0 标准

### 4.3 兼容性

| 平台 | 版本要求 |
|---|---|
| iOS | iOS 14 及以上 |
| Android | Android 8.0（API Level 26）及以上 |
| 网络环境 | 支持 4G/5G/Wi-Fi；弱网下打卡数据本地缓存，恢复网络后自动同步 |

### 4.4 数据与隐私

- 用户可随时导出自己的训练数据（CSV 格式）——P2 功能
- 用户可申请注销账号，注销后 30 天内数据可恢复，之后永久删除
- 遵守《个人信息保护法》，收集最小化原则

## 5. 附录

### 5.1 术语表

| 术语 | 定义 |
|---|---|
| 打卡 | 完成一次训练记录的行为，等同于「签到」 |
| 训练计划 | 用户预先设定的、包含多个训练日安排的结构化计划 |
| 训练日 | 计划中某一天要完成的训练内容（一组动作的集合） |
| 动作库 | 系统内置的健身动作数据库，包含名称、图示、肌群信息 |
| Streak（打卡流） | 连续打卡的天数，中断则归零 |
| 总容量 | 训练强度的量化指标，计算方式：Σ（重量 × 次数 × 组数） |
| 加油 | 好友间对打卡记录的正向互动，类似点赞功能 |

## 后续可细化方向

以下模块建议在 v0.1 评审通过后，进一步编写 IntentSpec 技术规范：

1. **动作库数据结构**：动作的分类体系、属性字段、图示资源管理方案
2. **好友动态 Feed 的隐私控制逻辑**：好友关系是单向还是双向？动态可见范围的边界规则
3. **打卡数据同步机制**：离线优先设计，冲突解决策略
4. **消息通知系统**：推送频率控制，防止骚扰用户的策略
5. **打卡海报生成**：前端生成还是服务端渲染？图片存储方案
