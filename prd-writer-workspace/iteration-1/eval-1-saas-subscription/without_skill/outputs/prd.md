# PRD：独立开发者订阅用户管理 SaaS 工具

**文档版本：** v1.0
**创建日期：** 2026-03-23
**状态：** 草稿

---

## 1. 产品概述

### 1.1 背景与问题

独立开发者（Indie Developer）在发布付费产品（SaaS、插件、工具等）后，往往面临以下痛点：

- 订阅用户数据分散在支付平台（Stripe、Paddle、LemonSqueezy 等）、邮件系统、数据库之间，缺乏统一视图。
- 无法方便地查看哪些用户即将到期、哪些用户存在付款失败问题。
- 手动发送欢迎邮件、续费提醒效率低，且容易遗漏。
- 缺乏针对独立开发者规模的轻量级工具，现有企业级 CRM 过于复杂昂贵。

### 1.2 产品定位

一款面向独立开发者的**轻量级订阅用户管理平台**，帮助开发者在一个地方统一管理其产品的订阅用户生命周期，包括：订阅状态追踪、到期管理、付款问题处理，以及自动化邮件触达。

### 1.3 目标用户

| 角色 | 描述 |
|------|------|
| 主要用户 | 独立开发者，有 1～5 个付费产品，订阅用户规模在 10～10,000 人之间 |
| 次要用户 | 小型团队（1～5 人）运营的 SaaS 产品负责人 |

### 1.4 产品目标

1. 让开发者 **5 分钟内**了解当前订阅用户的整体状况。
2. 将续费提醒、欢迎邮件等重复操作**自动化**，减少人工干预。
3. 降低因付款失败导致的**用户流失**（减少 churn）。

## 2. 核心功能需求

### 2.1 用户管理（Subscriber Management）

#### 2.1.1 订阅用户列表

- 展示所有订阅用户，支持分页、搜索（按邮箱、姓名）、筛选。
- 筛选条件：订阅状态（活跃 / 已取消 / 已过期 / 付款失败）、所属产品、到期时间范围。
- 列表字段：用户邮箱、姓名（可选）、订阅产品、订阅计划、订阅开始时间、到期时间、状态、最近付款状态。
- 支持批量操作：批量发邮件、批量导出。

#### 2.1.2 用户详情页

- 基本信息：邮箱、姓名、注册时间、来源渠道（如有）。
- 订阅历史：所有订阅记录，含计划变更、续费、取消记录。
- 付款历史：每笔付款记录（金额、时间、状态、失败原因）。
- 邮件发送记录：向该用户发送过的所有邮件列表。
- 备注：开发者可为用户添加内部备注。

#### 2.1.3 用户数据导入 / 同步

- 支持 CSV 手动导入（字段映射功能）。
- Webhook 接入：通过 Webhook 接收第三方支付平台的订阅事件，自动同步用户状态。
- 支持的支付平台（MVP 阶段）：Stripe、LemonSqueezy。
- API 接入：提供 REST API，允许开发者通过代码写入 / 更新用户数据。

### 2.2 订阅状态管理（Subscription Tracking）

#### 2.2.1 状态定义

| 状态 | 说明 |
|------|------|
| Active（活跃） | 订阅有效，下次续费未到期 |
| Expiring Soon（即将到期） | 距离到期日 ≤ N 天（可配置，默认 7 天） |
| Expired（已过期） | 到期日已过，未续费 |
| Cancelled（已取消） | 用户主动取消订阅 |
| Past Due（付款失败） | 自动续费失败，等待重试或处理 |
| Trialing（试用中） | 处于免费试用期 |

#### 2.2.2 到期提醒看板

- 专属视图：按到期时间升序展示"即将到期"和"已过期"用户。
- 快捷操作：一键发送续费提醒邮件给选中用户。

#### 2.2.3 付款问题看板

- 展示所有 Past Due 状态用户。
- 显示失败原因（如卡已过期、余额不足等，来自支付平台）。
- 快捷操作：发送付款提醒邮件、标记已处理。

### 2.3 邮件功能（Email Campaigns & Automation）

#### 2.3.1 邮件模板管理

- 内置模板：欢迎邮件、续费提醒、付款失败提醒、订阅取消确认。
- 支持自定义模板，使用富文本编辑器（支持 HTML）。
- 模板变量支持：`{{user_name}}`、`{{product_name}}`、`{{expiry_date}}`、`{{plan_name}}` 等动态字段。
- 模板预览功能（可预填测试数据查看效果）。

#### 2.3.2 手动发送邮件

- 从用户列表或详情页，选择模板后手动发送邮件给单个或多个用户。
- 发送前预览确认。
- 发送记录追踪（发送时间、状态：已发送 / 已送达 / 已打开 / 已点击 / 退信）。

#### 2.3.3 自动化邮件触发（Automation）

| 触发事件 | 默认动作 | 可配置项 |
|----------|----------|----------|
| 新用户订阅 | 发送欢迎邮件 | 延迟时间、使用模板 |
| 距到期 N 天 | 发送续费提醒 | N 值（默认 7 天）、模板 |
| 距到期 M 天（再次提醒） | 发送第二次续费提醒 | M 值（默认 1 天）、模板 |
| 付款失败 | 发送付款失败通知 | 延迟时间、模板 |
| 订阅取消 | 发送取消确认邮件 | 是否启用、模板 |
| 试用期结束前 | 发送转化提醒 | 提前天数、模板 |

- 自动化规则可单独启用 / 禁用。
- 自动化日志：记录每次自动发送的触发原因、时间、目标用户。

#### 2.3.4 邮件发送配置

- 支持接入开发者自有的邮件发送服务：
  - SMTP（自定义）
  - SendGrid
  - Resend
  - Mailgun
- 发件人域名需通过 DNS 验证（SPF / DKIM）。
- 平台提供默认共享发件域（用于快速体验，但建议生产环境使用自有域名）。

### 2.4 产品管理（Product Management）

- 开发者可在平台内创建多个"产品"，每个产品独立管理其订阅用户。
- 每个产品可配置多个"订阅计划"（如月付 / 年付 / 生命周期等）。
- 产品层面的统计：总订阅数、活跃订阅数、月度经常性收入（MRR）估算、流失率（Churn Rate）。

### 2.5 数据看板（Dashboard）

主页面提供整体概览：

- 关键指标卡片：活跃订阅总数、本月新增订阅、即将到期（7 天内）用户数、付款失败用户数、MRR 估算。
- 订阅趋势折线图（近 30 天 / 90 天）。
- 最近事件流：最新的订阅、取消、付款失败事件列表。
- 快捷入口：即将到期看板、付款问题看板。

## 3. 非功能性需求

### 3.1 性能

- 用户列表页加载时间 < 2 秒（数据量 ≤ 10,000 条）。
- 邮件发送（单封）响应时间 < 5 秒。
- 自动化邮件触发延迟 < 15 分钟（从事件发生到邮件发出）。

### 3.2 安全性

- 所有数据传输使用 HTTPS / TLS 1.2+。
- 用户密码使用 bcrypt 加密存储。
- 支付相关数据（如信用卡信息）不存储在本平台，仅存储支付平台返回的 ID 和状态。
- 支持双因素认证（2FA，TOTP 方式）。
- 数据隔离：多租户架构，开发者只能访问自己的数据。

### 3.3 可靠性

- 服务可用性目标：99.5% SLA（月度）。
- 邮件发送具备重试机制（失败后最多重试 3 次，间隔指数退避）。
- 关键数据（用户、订阅记录）每日自动备份。

### 3.4 可扩展性

- 系统设计支持单个开发者账户管理最多 100,000 名订阅用户。
- Webhook 事件处理使用异步队列，防止流量峰值导致丢失事件。

### 3.5 合规性

- 提供数据导出功能（CSV / JSON），方便开发者随时迁移数据。
- 支持用户数据删除请求（符合 GDPR 要求）。
- 邮件发送遵循 CAN-SPAM / GDPR 要求（含退订链接）。

## 4. 用户旅程

### 4.1 新开发者上手旅程

```
注册账号
  → 创建第一个产品
  → 选择订阅计划配置
  → 接入支付平台（配置 Webhook 或导入 CSV）
  → 配置邮件发送服务
  → 启用自动化邮件规则
  → 查看 Dashboard，确认数据同步正常
```

### 4.2 日常使用旅程

```
打开 Dashboard，查看关键指标
  → 处理"付款失败"看板中的问题用户（发邮件或标记处理）
  → 查看"即将到期"用户，确认提醒邮件已发出
  → 查看最新注册用户，确认欢迎邮件已触发
  → 按需手动发送邮件或查看邮件统计
```

## 5. 产品边界（不在 MVP 范围内）

以下功能在 v1.0 MVP 阶段不纳入，留待后续迭代：

| 功能 | 原因 |
|------|------|
| 内置支付收款功能 | 复杂度高，MVP 专注管理而非支付 |
| 客户支持工单系统 | 超出核心场景 |
| 高级分析报表（群组分析、LTV 计算） | 后续增长阶段再引入 |
| 移动端 App | Web 优先，移动端后续规划 |
| 多语言界面 | MVP 先做英文版 |
| 社交登录（Google / GitHub OAuth） | 可快速迭代加入 |
| Slack / Discord 通知集成 | 后续扩展 |

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 支付平台 Webhook 数据不稳定 | 用户状态同步不及时 | 支持手动刷新 + 定期轮询补偿机制 |
| 邮件进入垃圾箱 | 自动化邮件效果差 | 提供 DNS 验证指引，使用声誉好的发件 IP |
| 开发者数据隐私顾虑 | 用户不敢接入 | 提供数据导出、删除、清晰隐私政策 |
| 竞品（如 Baremetrics、ChartMogul）功能重叠 | 差异化不足 | 聚焦独立开发者场景，强调轻量、易用、低价 |
| 自动化邮件被滥用（垃圾邮件） | 影响平台发件域名声誉 | 限制发送频率，审查账号，强制包含退订链接 |

## 10. 附录

### 10.1 关键术语

| 术语 | 说明 |
|------|------|
| MRR | Monthly Recurring Revenue，月度经常性收入 |
| Churn Rate | 流失率，一段时间内取消订阅的用户比例 |
| Webhook | 第三方服务向本平台推送事件的 HTTP 回调机制 |
| Past Due | 自动续费失败，待处理的付款状态 |
| Automation | 基于事件自动触发邮件发送的规则引擎 |

### 10.2 参考竞品

- **Baremetrics**：专注 SaaS 指标分析，偏重数据分析，邮件功能弱。
- **ChartMogul**：企业级，价格高，不适合独立开发者。
- **Paddle Retain**：仅限 Paddle 用户，绑定性强。
- **Mailchimp**：通用邮件营销，不具备订阅状态管理能力。

本产品的差异化在于：**专为独立开发者设计、轻量易用、多支付平台兼容、订阅状态与邮件功能深度整合**。
