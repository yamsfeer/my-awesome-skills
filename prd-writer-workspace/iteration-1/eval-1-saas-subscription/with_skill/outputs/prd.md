## 1. 产品愿景

### 1.1 产品定位

SubManager 是一款专为独立开发者设计的轻量级订阅用户管理平台，帮助开发者在一个地方清晰掌握订阅状态、识别付款风险、并通过自动化邮件与用户保持沟通，从而降低用户流失、提升续费率。

### 1.2 核心价值主张

- **一目了然的订阅全貌**：实时查看所有订阅用户的状态（活跃 / 即将到期 / 已取消 / 付款失败），告别手动查 Stripe 后台的低效操作
- **付款风险提前感知**：自动识别付款失败、信用卡即将到期等高危用户，提前干预挽回收入
- **自动化邮件触达**：内置欢迎邮件、续费提醒、付款失败提醒等邮件模板，一次配置、持续运转
- **极简上手，无需运维**：面向独立开发者，无需配置服务器，接入支付平台（Stripe 等）后 5 分钟内可用
- **多产品聚合管理**：支持在同一账号下管理多个产品的订阅数据，独立开发者多项目并行时尤为实用

### 1.3 差异化优势

| 对比维度 | SubManager | Stripe 原生后台 | 自建脚本/表格 |
|----------|------------|----------------|--------------|
| 订阅状态全览 | ✅ 统一仪表盘 | ❌ 分散在多个页面 | ❌ 需手动维护 |
| 付款风险预警 | ✅ 自动识别 + 通知 | ⚠️ 有数据但不主动提醒 | ❌ 无 |
| 自动化邮件 | ✅ 内置模板 + 规则引擎 | ❌ 不支持 | ❌ 需自己开发 |
| 多产品管理 | ✅ 原生支持 | ⚠️ 需多账号切换 | ❌ 难以扩展 |
| 上手成本 | ✅ 极低（API Key 接入） | — | ❌ 高（需开发） |
| 价格 | 💰 面向个人的低价 SaaS | 免费但功能有限 | 时间成本高 |

### 1.4 成功指标（6 个月目标）

| 指标 | 目标值 |
|------|--------|
| 注册用户数 | 500+ |
| 付费用户数 | 100+ |
| 免费→付费转化率 | ≥ 20% |
| 月续费留存率（MRR Retention） | ≥ 85% |
| 平均每用户管理订阅数 | ≥ 50 条 |
| 邮件发送成功率 | ≥ 98% |

## 2. 目标用户

### 2.1 核心用户画像

| 维度 | 描述 |
|------|------|
| 职业 | 独立开发者（indie hacker）、个人 SaaS 创始人、副业开发者 |
| 年龄 | 25–40 岁 |
| 地域 | 全球英语区为主，中文区次之 |
| 收入 | 产品月收入 $500–$5000（MRR 早期阶段） |
| 技术能力 | 具备基本编程能力，熟悉 API、Webhook 概念 |
| 工具偏好 | 使用 Stripe / Paddle 作为支付平台 |
| 痛点敏感度 | 对时间成本极为敏感，不愿花时间在"非产品本身"的管理工作上 |

### 2.2 典型使用场景

**场景 A：新用户付款后忘记发欢迎邮件**

小明是一位独立开发者，他的工具类 SaaS 每周能新增 10–20 名付费用户。他发现用户付款后经常没有引导，导致用户不知道如何使用产品功能，流失率偏高。他希望有一个系统能在用户付款后 1 分钟内自动发出欢迎邮件并附上入门指引，而他完全不需要手动操作。

**场景 B：月底批量续费前的风险排查**

小红管理着两款 SaaS 产品，每月月底会有一批年费用户到期续费。她担心有用户的信用卡已经过期或余额不足，想在续费日前 7 天提前给这些用户发提醒邮件，同时在仪表盘上高亮显示"高风险"用户，方便她手动跟进。

**场景 C：查看某个用户的订阅历史**

小张收到一封来自用户的投诉邮件，说他明明已经续费但系统显示未订阅。小张需要快速查到该用户的订阅记录、付款历史和邮件发送记录，在 2 分钟内定位问题原因并回复用户。

### 2.3 当前痛点

| 痛点 | 现有解决方式 | 不满之处 |
|------|-------------|---------|
| 订阅状态散落在 Stripe 后台 | 手动逐个查看 | 效率低，无法批量筛选 |
| 没有付款失败的主动提醒 | 等 Stripe 发系统邮件 | 邮件样式差，无品牌感，且只通知用户不通知开发者 |
| 欢迎邮件/续费提醒需要自己写代码 | 自建 Webhook + 邮件服务 | 开发成本高，维护麻烦 |
| 多产品订阅数据无法聚合 | 不同账号来回切换 | 认知负担重，容易遗漏 |

## 3. 功能需求

用户活动（主干）：**接入产品** → **查看订阅数据** → **识别风险用户** → **邮件触达用户** → **账号与产品管理**

---

### 3.1 接入产品

#### 用户任务 1.1：连接支付平台

**用户目标：** 将自己的 Stripe 账号数据同步到 SubManager，使订阅数据可以被统一管理。

**系统功能：**
- 支持通过 Stripe API Key（Restricted Key）完成授权接入，无需 OAuth 跳转
- 接入后自动拉取历史订阅数据（最近 12 个月）
- 支持设置 Webhook，实时同步新增订阅、取消、付款失败等事件
- 接入成功后展示数据同步进度和状态

**优先级：** P0

**验收标准：**
- 用户填入 API Key 后，系统在 30 秒内完成连接验证并开始同步
- 历史数据同步完成时，在界面给出明确提示
- API Key 无效时，给出清晰的错误提示和修复建议

---

#### 用户任务 1.2：创建和管理产品

**用户目标：** 在 SubManager 中区分管理自己的多个产品，避免数据混淆。

**系统功能：**
- 支持创建多个"产品"，每个产品关联一个或多个 Stripe Product/Price
- 每个产品可设置名称、描述、图标
- 产品列表展示每个产品的订阅总数、MRR、活跃率概览

**优先级：** P0

---

### 3.2 查看订阅数据

#### 用户任务 2.1：查看订阅用户列表

**用户目标：** 快速掌握所有订阅用户的当前状态，筛选出需要关注的用户群体。

**系统功能：**
- 展示订阅用户列表，包含字段：用户邮箱、姓名、订阅计划、订阅状态、开始时间、到期时间、金额
- 支持按订阅状态筛选：活跃（Active）/ 试用中（Trial）/ 即将到期（Expiring Soon，7 天内）/ 已过期（Expired）/ 付款失败（Payment Failed）/ 已取消（Canceled）
- 支持按邮箱、姓名搜索用户
- 支持按到期时间、订阅金额排序
- 支持导出当前列表为 CSV

**优先级：** P0

**验收标准：**
- 列表首次加载时间 < 1.5s（数据量 < 1000 条）
- 筛选和排序操作实时响应，无需刷新页面

---

#### 用户任务 2.2：查看单个用户详情

**用户目标：** 快速了解某个用户的完整订阅历史和沟通记录，处理用户反馈或核查问题。

**系统功能：**
- 用户详情页展示：基本信息（邮箱、注册时间）、当前订阅状态、历史订阅记录（含金额、状态、时间段）、付款记录（成功/失败/退款）、已发送邮件记录
- 支持在用户详情页手动触发发送邮件
- 展示用户在 Stripe 中的 Customer ID，方便跳转到 Stripe 后台

**优先级：** P0

---

#### 用户任务 2.3：查看数据概览仪表盘

**用户目标：** 通过关键指标一眼了解产品订阅业务的健康状况。

**系统功能：**
- 仪表盘展示核心指标：
  - MRR（月经常性收入）及环比变化
  - 活跃订阅用户数
  - 本月新增订阅数
  - 本月流失订阅数（Churn）
  - 付款失败用户数（高亮红色预警）
  - 7 天内到期用户数
- 支持按产品维度切换查看

**优先级：** P0

---

### 3.3 识别风险用户

#### 用户任务 3.1：查看付款失败用户

**用户目标：** 快速找到付款失败的用户，评估收入损失，并决定是否手动介入。

**系统功能：**
- 仪表盘和列表页均突出显示付款失败用户
- 付款失败用户卡片展示：失败时间、失败原因（余额不足 / 卡号过期 / 银行拒绝等）、失败金额、Stripe 重试计划
- 支持一键触发"付款失败提醒邮件"给该用户
- 支持标记用户为"已手动处理"，避免重复操作

**优先级：** P0

---

#### 用户任务 3.2：查看即将到期用户

**用户目标：** 提前识别即将到期的用户，在到期前发送续费提醒，降低被动流失。

**系统功能：**
- 支持筛选"X 天内到期"的用户（可自定义天数：3 / 7 / 14 / 30 天）
- 支持批量选择即将到期用户并发送续费提醒邮件
- 展示该批用户的合计到期金额，帮助评估收入风险

**优先级：** P0

---

### 3.4 邮件触达用户

#### 用户任务 4.1：配置邮件发送服务

**用户目标：** 连接自己的邮件发送服务，确保邮件能以自定义域名发出，具备品牌感。

**系统功能：**
- 支持接入以下邮件服务：Resend、SendGrid、Mailgun（通过 API Key）
- 支持配置发件人名称和发件邮箱地址
- 支持发送测试邮件验证配置是否正确

**优先级：** P0

---

#### 用户任务 4.2：管理邮件模板

**用户目标：** 编辑邮件内容，使邮件符合自己产品的品牌风格和沟通口吻。

**系统功能：**
- 内置以下场景的默认邮件模板：
  - 新订阅欢迎邮件
  - 续费成功通知
  - 续费提醒（到期前 7 天）
  - 付款失败通知
  - 订阅取消确认
- 支持通过富文本编辑器自定义每个模板的主题、正文内容
- 支持在邮件中使用变量占位符：`{{user_name}}`、`{{plan_name}}`、`{{expiry_date}}`、`{{amount}}` 等
- 支持预览邮件渲染效果

**优先级：** P0

**验收标准：**
- 占位符在预览时正确替换为示例数据
- 编辑后的模板实时保存，不丢失内容

---

#### 用户任务 4.3：配置自动化邮件规则

**用户目标：** 设置触发条件，让系统在特定事件发生时自动发送对应邮件，无需手动操作。

**系统功能：**
- 支持创建自动化规则，每条规则包含：触发事件 + 延迟时间 + 邮件模板
- 支持的触发事件：
  - 用户新增订阅（立即 / 延迟 N 小时）
  - 订阅到期前 N 天
  - 付款失败（立即 / 延迟 N 小时重试后仍失败）
  - 订阅取消
- 支持为每条规则单独开启/关闭
- 展示每条规则的历史触发次数和发送成功率

**优先级：** P0

**验收标准：**
- 规则触发后，邮件在 5 分钟内发出
- 规则关闭后，该规则不再触发任何邮件

---

#### 用户任务 4.4：手动发送邮件

**用户目标：** 针对特定用户或用户群组，手动发送一次性邮件（如公告、特别优惠）。

**系统功能：**
- 支持在用户列表页批量选中用户，发送选定的邮件模板
- 支持在单个用户详情页直接发送邮件
- 发送前展示"预计发送给 X 位用户"的确认弹窗

**优先级：** P1

---

#### 用户任务 4.5：查看邮件发送记录

**用户目标：** 了解邮件发送情况，核查某封邮件是否成功发出。

**系统功能：**
- 邮件日志列表展示：收件人、邮件主题、发送时间、发送状态（发送中 / 已发送 / 失败）
- 支持按用户邮箱、邮件类型、状态筛选
- 发送失败时展示失败原因，支持手动重试

**优先级：** P1

---

### 3.5 账号与产品管理

#### 用户任务 5.1：注册与登录

**用户目标：** 快速创建账号并登录，开始使用产品。

**系统功能：**
- 支持邮箱 + 密码注册
- 支持 Google OAuth 一键登录
- 支持邮箱验证（注册后发送验证邮件）
- 支持密码找回

**优先级：** P0

---

#### 用户任务 5.2：管理账号订阅计划（SubManager 自身的订阅）

**用户目标：** 查看并管理自己使用 SubManager 的订阅计划，必要时升级或取消。

**系统功能：**
- 展示当前套餐及使用量（如：已管理订阅数 / 计划上限）
- 支持查看历史账单
- 支持升级套餐
- 免费套餐：管理 ≤ 100 条订阅、发送 ≤ 500 封邮件/月

**优先级：** P1

## 4. 非功能需求

### 4.1 性能要求

| 指标 | 要求 |
|------|------|
| 页面首次加载（含数据） | P95 < 1.5s |
| 列表筛选/搜索响应 | P95 < 300ms |
| 邮件自动化触发延迟 | < 5 分钟 |
| Stripe Webhook 处理延迟 | < 30 秒 |
| 系统可用性 | ≥ 99.5% / 月 |

### 4.2 安全要求

- Stripe API Key 存储时必须加密（AES-256），不以明文存入数据库
- 邮件服务 API Key 同上
- 用户认证使用 JWT，Token 有效期 7 天，支持主动登出使 Token 失效
- 所有接口通过 HTTPS 传输
- 不同用户的数据严格隔离（多租户），禁止跨账号访问

### 4.3 兼容性

| 平台 | 要求 |
|------|------|
| 桌面浏览器 | Chrome 100+、Firefox 100+、Safari 15+、Edge 100+ |
| 移动端 | 响应式布局，支持手机浏览器基本查看（非核心场景） |
| 屏幕分辨率 | 1280×800 及以上完整支持 |

## 5. 附录

### 5.1 术语表

| 术语 | 说明 |
|------|------|
| MRR | Monthly Recurring Revenue，月经常性收入，衡量订阅业务规模的核心指标 |
| Churn | 用户流失，指订阅取消或未续费 |
| Trial | 试用期订阅，通常为免费试用 N 天后自动转为付费 |
| Webhook | 事件通知机制，Stripe 在发生付款、取消等事件时主动推送数据到指定 URL |
| Restricted Key | Stripe 提供的受限 API Key，可只授予只读或特定权限，安全性更高 |
| P0 / P1 / P2 | 功能优先级：P0 为 MVP 必须有，P1 为二期重要功能，P2 为后续增值功能 |

## 后续可深化方向

以下功能模块建议在 v0.1 评审通过后，进一步编写 IntentSpec 技术规范：

1. **Stripe Webhook 处理逻辑**：事件类型的完整映射、幂等性处理、重试机制
2. **邮件自动化规则引擎**：规则冲突处理、发送频率限制（防止骚扰用户）
3. **多租户数据隔离方案**：数据库设计、权限中间件
4. **免费套餐限制策略**：限额触发时的用户体验和升级引导流程
