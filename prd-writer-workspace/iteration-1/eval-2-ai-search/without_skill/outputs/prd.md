## 1. 背景与目标

### 1.1 背景

当前企业知识库产品的搜索功能基于关键词匹配（如 Elasticsearch 全文检索），存在以下痛点：

- **表达方式受限**：用户必须将问题拆解为关键词才能搜索，认知成本高。例如"上周财务部门提交了哪些审批"需要被拆成"财务 审批 2026-03"才能勉强检索。
- **语义理解缺失**：同义词、近义词无法匹配，如"审批"与"审核"、"季度汇报"与"Q1 报告"无法互相召回。
- **无直接答案**：搜索结果为文档列表，用户仍需手动翻阅，查找效率低。
- **时间、部门等结构化条件处理弱**：用户提问中隐含的时间范围、组织范围等无法被有效解析。

随着大语言模型（LLM）技术成熟，通过自然语言理解 + 知识检索增强生成（RAG）技术，可以让用户直接用日常语言提问，系统即时给出聚合性答案，并附带来源引用，大幅提升信息获取效率。

### 1.2 目标

| 目标维度 | 具体目标 |
|---|---|
| 用户体验 | 用户无需思考关键词，直接用自然语言提问即可得到答案 |
| 效率提升 | 从提问到获得答案的时间从平均 3 分钟降低至 30 秒以内 |
| 覆盖场景 | 支持时间范围、部门范围、文档类型等条件的复合查询 |
| 可信度 | 每条 AI 答案必须附带原始来源文档引用，可追溯可核实 |
| 安全合规 | AI 搜索严格遵守现有知识库的文档权限体系，不泄露越权内容 |

### 1.3 成功指标（Success Metrics）

- **搜索满意率**（用户对答案点赞 / 总 AI 搜索次数）≥ 70%（上线后 3 个月内）
- **功能渗透率**（使用过 AI 搜索的 DAU / 总 DAU）≥ 40%（上线后 3 个月内）
- **平均首次响应时间** ≤ 5 秒（P95）
- **搜索降级率**（AI 无法处理而退回关键词搜索的比例）≤ 15%

## 2. 用户与场景

### 2.1 目标用户

| 用户角色 | 描述 | 使用频率 |
|---|---|---|
| 普通员工 | 查找公司规章、流程文档、历史记录 | 高频 |
| 部门管理者 | 汇总部门内文件、追踪审批/汇报记录 | 中频 |
| HR / 法务 / 财务 | 查找特定合规文档、历史合同、制度文件 | 高频 |
| 知识库管理员 | 验证索引质量、监控搜索效果 | 低频 |

### 2.2 典型用户场景（User Stories）

#### 场景 1：跨时间范围的部门记录查询

> **角色**：财务部门主管
> **行为**：在搜索框输入「上周财务部门提交了哪些审批」
> **期望**：系统理解"上周"为相对时间范围，"财务部门"为组织筛选条件，"审批"为文档类型，直接返回汇总答案并列出相关审批单链接。

#### 场景 2：流程与制度查询

> **角色**：新入职员工
> **行为**：输入「报销流程是怎样的，最多能报多少」
> **期望**：系统从报销制度文档中提取关键步骤和金额上限，以结构化方式呈现答案，并标注来源文档。

#### 场景 3：历史会议纪要查询

> **角色**：项目经理
> **行为**：输入「上个季度产品团队关于 APP 改版讨论了哪些决策」
> **期望**：系统检索相关会议纪要，提炼出决策要点，按时间排序展示。

#### 场景 4：无结果兜底

> **角色**：任意员工
> **行为**：输入「公司明年准备上市吗」（知识库中无相关内容）
> **期望**：系统明确告知未找到相关文档，建议联系相关负责人，而非捏造答案。

### 3.2 自然语言输入与意图解析

#### 3.2.1 输入框升级

- 搜索框支持输入长句（最长 500 字符），placeholder 引导语改为「请直接提问，如：上周财务部门提交了哪些审批？」
- 输入框右侧提供「AI 搜索」与「关键词搜索」切换 Tab，默认为 AI 搜索。
- 支持键盘 `Enter` 提交，支持点击搜索按钮提交。

#### 3.2.2 意图解析

系统在执行检索前，先对用户输入进行意图解析，识别以下维度的信息：

| 解析维度 | 示例 | 说明 |
|---|---|---|
| 时间范围 | 「上周」「本月」「2025年Q3」 | 转换为具体日期区间 |
| 组织范围 | 「财务部门」「技术团队」 | 映射到企业组织架构节点 |
| 文档类型 | 「审批」「合同」「会议纪要」 | 映射到知识库文档分类标签 |
| 核心问题 | 「提交了哪些」「流程是怎样的」 | 提炼用户核心意图 |
| 作者/提交人 | 「张三提交的」 | 映射到用户账号 |

解析结果以结构化 JSON 传递给检索模块，同时在 UI 上以 Tag 形式展示解析到的条件（供用户确认或修改）。

#### 3.2.3 歧义处理

- 若解析到多个可能的意图（如"财务"可能是部门也可能是文档分类），系统选择置信度最高的，并在结果页提示「您是否在查找…」供用户确认。
- 若输入过于模糊（如仅输入「报告」），系统在给出检索结果的同时，提示用户补充更多条件。

---

### 3.3 语义检索与 RAG 答案生成

#### 3.3.1 向量索引构建

- 知识库中所有文档在入库时自动进行向量化（Embedding），存入向量数据库。
- 支持增量更新：文档新增、修改、删除时，相应向量索引实时（或准实时，延迟 ≤ 5 分钟）更新。
- 文档分块策略：按段落或语义边界切分，每块不超过 512 Token，块间保留 50 Token 重叠以保证上下文连续。

#### 3.3.2 混合检索

采用「向量检索 + 关键词检索」双路召回，融合排序：

1. **向量检索**：使用用户问题的 Embedding 在向量库中检索 Top-K（K=20）语义相似文档块。
2. **关键词检索**：基于现有 Elasticsearch 全文检索召回 Top-K 结果。
3. **融合排序（RRF / 重排模型）**：将两路结果融合并重排，取 Top-N（N=5~10）作为上下文传递给 LLM。

权限过滤在检索阶段强制执行（见 5.1 节）。

#### 3.3.3 答案生成（LLM）

- 将检索到的文档片段 + 用户问题组装成 Prompt，调用 LLM 生成自然语言答案。
- LLM 必须被约束为「仅基于提供的文档内容回答，不得使用外部知识编造内容」。
- 若检索结果与问题相关性不足，LLM 应返回「未找到相关内容」而非强行生成。
- 答案长度：简洁模式下不超过 300 字；详细模式下不超过 800 字（用户可切换）。

---

### 3.4 答案展示与来源引用

#### 3.4.1 答案卡片

搜索结果页顶部展示 AI 答案卡片，包含：

- **答案正文**：LLM 生成的自然语言答案，支持 Markdown 渲染（列表、加粗等）。
- **来源引用标注**：答案中关键事实对应的来源以上标数字标注（如 `[1]`）。
- **来源文档列表**：卡片底部列出所有引用的原始文档，展示文档标题、所属部门、创建时间，可点击跳转。
- **生成时间**：标注答案生成时间，提示信息可能不是最新状态。

#### 3.4.2 传统结果列表

AI 答案卡片下方保留传统文档列表（关键词高亮），供用户深入浏览。

#### 3.4.3 答案置信度提示

- 当 LLM 判断答案把握度较低时，在答案卡片顶部展示黄色提示「此答案基于部分匹配内容生成，建议核实原文」。
- 当检索结果为空时，展示「未在知识库中找到相关内容」，并提供关键词搜索入口。

---

### 3.5 多轮对话追问

- 用户可在答案卡片下方的输入框继续追问，系统维持上下文（会话 ID），支持最多 10 轮对话。
- 追问示例：首问「上周财务审批有哪些」→ 追问「其中金额超过 10 万的有几条」。
- 会话超时：30 分钟无操作后会话重置，下次提问作为新会话处理。
- 用户可主动点击「开始新对话」清除上下文。

---

### 3.6 反馈与纠错机制

- 每条 AI 答案卡片提供「赞」「踩」按钮。
- 点击「踩」后弹出可选原因：答案不准确 / 引用来源有误 / 内容不完整 / 其他。
- 反馈数据实时入库，供后台分析和模型迭代使用。
- 管理员可在后台查看负反馈率排行，定位高频问题查询。

---

### 3.7 搜索降级与兜底策略

| 触发条件 | 降级策略 |
|---|---|
| LLM 服务超时（> 10 秒）| 自动降级为关键词搜索，页面提示「AI 搜索暂时不可用，已为您展示关键词结果」|
| 向量服务不可用 | 仅用关键词检索结果生成答案（或直接降级） |
| 检索结果为空 | 展示「未找到相关文档」，引导用户换关键词或联系管理员 |
| 问题包含敏感词 | 拒绝回答，提示用户联系相关负责人 |

---

### 3.8 管理后台：搜索监控与配置

#### 3.8.1 监控面板

- 展示 AI 搜索 PV/UV、平均响应时间、满意率（点赞率）、降级率。
- 提供高频问题排行榜，展示最近 7/30 天 Top 20 查询问题。
- 展示无结果查询 Top 20，辅助知识库内容补充决策。

#### 3.8.2 索引管理

- 支持手动触发全量重建向量索引。
- 支持查看各文档的向量化状态（待处理 / 处理中 / 完成 / 失败）。
- 支持配置哪些文档分类参与 AI 搜索索引。

#### 3.8.3 模型与参数配置

- 支持配置 LLM 模型版本（如切换不同供应商或版本）。
- 支持配置答案最大 Token 数、检索 Top-K 等参数。
- 支持配置敏感词列表。

## 4. 非功能需求

### 4.1 性能

| 指标 | 要求 |
|---|---|
| AI 搜索端到端响应时间（P50）| ≤ 3 秒 |
| AI 搜索端到端响应时间（P95）| ≤ 8 秒 |
| 向量索引更新延迟 | ≤ 5 分钟 |
| 系统并发支持 | 支持 200 QPS AI 搜索请求 |
| 系统可用性 | ≥ 99.5%（AI 搜索模块独立故障不影响关键词搜索） |

### 4.2 安全

- AI 搜索结果严格遵守知识库现有权限体系，用户只能检索到自己有权限查看的文档内容。
- 用户搜索问题与 AI 生成答案日志加密存储，保留 180 天。
- LLM API 调用走内网或专线，禁止明文传输敏感企业数据。
- 禁止 AI 答案中包含其他用户的个人隐私信息（如手机号、身份证等）。

### 4.3 可靠性

- AI 搜索模块与关键词搜索模块解耦部署，AI 模块故障时自动降级。
- 向量数据库与主业务数据库分离部署。

### 4.4 可扩展性

- 向量索引方案支持千万级文档块规模扩展。
- LLM 供应商支持可配置化切换（避免单一供应商锁定）。

## 5. 数据与权限

### 5.1 权限控制

- **检索层权限过滤**：向量检索与关键词检索均在用户身份上下文下执行，过滤掉用户无权访问的文档。
- **答案生成层验证**：传递给 LLM 的文档片段必须全部来自用户有权访问的内容，不得混入越权内容。
- **引用链接权限**：来源文档链接点击时，仍走原有文档权限校验，无权限则提示「无权访问」。

### 5.2 数据流

```
用户提问
   ↓
意图解析（含时间/部门/类型提取）
   ↓
混合检索（向量检索 + 关键词检索，权限过滤）
   ↓
重排与上下文裁剪
   ↓
LLM 答案生成（Prompt 含文档片段 + 问题）
   ↓
答案后处理（来源标注、敏感词过滤）
   ↓
结果返回给用户
```

### 5.3 数据存储

| 数据类型 | 存储方案 | 保留策略 |
|---|---|---|
| 文档向量 | 向量数据库（如 Milvus / Qdrant） | 随文档生命周期 |
| 用户搜索日志 | 加密日志库 | 180 天 |
| 反馈数据 | 关系型数据库 | 永久保留 |
| 多轮对话上下文 | Redis（内存） | 30 分钟 TTL |

## 7. 验收标准

### 7.1 功能验收

| 验收项 | 验收标准 |
|---|---|
| 自然语言搜索 | 输入「上周财务部门提交了哪些审批」可返回有效答案 |
| 时间解析 | 「上周」「本月」「上个季度」等相对时间正确解析为日期区间 |
| 部门/人员过滤 | 指定部门/人员的查询只召回对应范围文档 |
| 权限隔离 | 用户 A 无法通过 AI 搜索获取用户 A 无权查看的内容 |
| 来源引用 | 答案中每个关键事实有对应来源标注，来源文档链接可正常跳转 |
| 无结果兜底 | 知识库无相关内容时，明确告知而非编造答案 |
| 降级机制 | LLM 超时时自动降级为关键词搜索，不影响用户使用 |
| 多轮追问 | 支持连续追问，上下文保持 10 轮内连贯 |
| 反馈功能 | 点赞/踩可正常提交，踩后原因选项可选 |
| 管理后台 | 可查看 AI 搜索 PV、满意率、高频问题列表 |

### 7.2 性能验收

| 验收项 | 标准 |
|---|---|
| P50 响应时间 | ≤ 3 秒 |
| P95 响应时间 | ≤ 8 秒 |
| 并发 50 用户 | 响应时间无明显劣化（P95 ≤ 10 秒）|
| 向量索引更新 | 文档更新后 5 分钟内可被 AI 搜索检索到 |

### 7.3 安全验收

| 验收项 | 标准 |
|---|---|
| 越权访问测试 | 构造 10 组权限隔离测试用例，全部通过 |
| 敏感词过滤 | 配置敏感词后，相关问题触发拦截提示 |

## 8. 风险与依赖

### 8.1 主要风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|---|---|---|---|
| LLM 答案幻觉（编造内容）| 中 | 高 | Prompt 严格约束 + 来源引用强制展示 + 用户可核实原文 |
| 向量索引与权限不同步 | 低 | 高 | 检索层双重权限过滤，索引不承载权限信息 |
| LLM API 成本超预期 | 中 | 中 | 设置每日调用量上限；缓存高频相同问题的答案 |
| 用户输入敏感信息被 LLM 处理 | 低 | 高 | 敏感词检测 + 日志脱敏存储 + 与供应商签署数据协议 |
| 响应时间不满足预期 | 中 | 中 | 流式输出降低感知；异步预热常用文档向量 |
