## 案例 1：电商 APP（闪电购）

### 背景
产品经理小李受命打造一款面向下沉市场年轻女性的短视频电商 APP。

### 阶段 1：机会发现

**数据来源：**
- 用户访谈：20 位二三线城市女性用户
- 竞品分析：淘宝、拼多多、抖音电商
- 数据分析：下沉市场消费趋势报告

**发现的问题：**

| 问题 | 严重程度 | 现有解决方案不足 |
|------|----------|------------------|
| 商品图片与实物差距大 | 高 | 传统电商图文展示不够直观 |
| 价格便宜但质量没保障 | 高 | 缺乏工厂直供信任机制 |
| 退货流程麻烦 | 中 | 需要寄回商品，等待退款 |
| 决策成本高 | 中 | 需要看评价、比价格 |

**关键洞察：**
- 下沉市场用户更相信"眼见为实"
- 价格敏感度高，但愿意为"所见即所得"付费
- 对退货麻烦极度反感

### 阶段 2：机会评估

**RICE 评分：**

| 因素 | 评分 | 说明 |
|------|------|------|
| Reach（触达）| 9/10 | 下沉市场 6 亿用户 |
| Impact（影响）| 8/10 | 解决核心购物痛点 |
| Confidence（信心）| 7/10 | 有数据支撑 |
| Effort（成本）| 6/10 | 中等复杂度 |
| **RICE 总分** | **84** | |

**市场规模：**
- TAM：中国电商市场 3 万亿
- SAM：下沉市场电商 8000 亿
- SOM：第一年目标 1% 份额 = 80 亿

**差异化定位：**
> 面向二三线城市年轻女性的短视频电商平台，让工厂好货直达消费者。

### 阶段 3：用户研究

**Persona：小丽**
- 28 岁，县城服装店店员，月收入 4000 元
- 每天晚上刷抖音，看到喜欢的会买
- 痛点：买的东西经常和想象的不一样
- 目标：买到物美价廉、真实可靠的商品

**JTBD：**
> 当我想要买一件衣服时，我想看到真实的展示效果，以便确定是否适合我。

**用户场景：**
- 场景 A：晚上 8-10 点刷短视频，看到喜欢的直接下单
- 场景 B：周末想买某类商品，搜索对比后下单

### 阶段 4：解决方案设计

**Kano 模型分类：**

| 功能 | 分类 | 说明 |
|------|------|------|
| 商品展示 | 基本需求 | 必须有 |
| 短视频播放 | 兴奋需求 | 超出期待 |
| 30 秒退款 | 兴奋需求 | 差异化亮点 |
| 工厂直供价 | 期望需求 | 越好越满意 |

**MoSCoW 优先级：**

**Must Have：**
- 视频播放、推荐列表、商品详情
- 购物车、地址管理、支付系统
- 订单管理、确认收货

**Should Have：**
- 搜索功能、物流查询、售后服务

**Could Have：**
- 评价系统、会员系统

**Won't Have（V1）：**
- 直播带货、社交电商

**MVP 定义（SMART）：**
- S：短视频展示 + 基础交易闭环
- M：下单转化率 ≥ 8%，30 天留存 ≥ 40%
- A：团队 10 人，3 个月可完成
- R：直接解决"所见即所得"痛点
- T：12 周发布内测版

### 阶段 5：PRD 文档

**用户旅程地图：**

```
┌─────────────────────────────────────────────────────────────┐
│ 用户活动   │  浏览发现   │  下单购买   │  收货售后  │  再次购买  │
├───────────┼─────────────┼─────────────┼────────────┼───────────┤
│ 用户任务   │ 看短视频    │ 选规格加购  │ 查看物流   │ 查看历史  │
│           │ 浏览推荐    │ 填写地址    │ 确认收货   │ 再次下单  │
│           │ 搜索商品    │ 发起支付    │ 申请售后   │ 推荐复购  │
│           │ 查看详情    │             │ 评价商品   │           │
├───────────┼─────────────┼─────────────┼────────────┼───────────┤
│ 功能点    │ 视频播放    │ 购物车      │ 物流查询   │ 订单列表  │
│           │ 推荐算法    │ 地址管理    │ 消息推送   │ 智能推荐  │
│           │ 搜索引擎    │ 支付系统    │ 退款系统   │ 会员系统  │
│           │ 商品详情    │             │ 评价系统   │           │
└───────────┴─────────────┴─────────────┴────────────┴───────────┘
```

**核心用户故事：**

**US-001：看短视频**
> 作为小丽，我想通过短视频直观了解商品，以便做出购买决策。

**AC：**
- Given 用户打开 APP
- When 用户上下滑动
- Then 自动播放下一个短视频
- And 视频下方显示关联商品卡片

**优先级：** P0
**对应 IntentSpec：** video-player-intentspec.md

**非功能需求：**
- 首页加载时间 P95 < 1.5s
- 视频播放首帧时间 < 500ms
- 系统可用性 > 99.9%
- 支持并发用户 10 万

## 案例 2：支付功能 IntentSpec

### 背景
支付是电商 APP 的核心交易环节，失败代价极高。以下是支付功能的完整 IntentSpec。

---

# IntentSpec：支付系统

## 1. 目标（Goal）

### 观察到的现象
- 过去 30 天数据显示，发起支付的用户中有 18% 未完成交易
- 其中 11% 因支付失败后无明确引导而直接离开
- 4% 因支付加载超过 3 秒而放弃
- 3% 因不确定支付是否成功而重复提交导致订单异常

### 影响评估
- 每周约损失 ¥86,000 交易额（约 430 笔订单）
- 客服每周处理重复支付投诉 32 单

### 依据来源
- 用户访谈（5人）：失败时不知道发生了什么，不敢重试
- 热力图：支付失败页面 73% 用户在 5 秒内关闭
- 数据监控：支付接口 P95 响应时间 4.2 秒

## 2. 用户目标（User Job）

作为下单用户，我希望：
1. 在支付遇到问题后，清楚知道发生了什么以及如何继续
2. 在 2 次点击内完成支付重试或更换支付方式
3. 在支付过程中随时知晓当前状态，避免重复支付
4. 支付完成后能立即看到明确的 success/fail 状态

## 3. 成果（Outcomes）

### 业务指标

| 指标 | 当前值 | 目标值 | 验证方式 |
|------|--------|--------|----------|
| 支付完成率 | 82% | ≥ 91% | 支付成功事件 / 发起支付事件 |
| 支付失败后 5 分钟内重试率 | 12% | ≥ 45% | 追踪失败会话的后续行为 |
| 重复支付引发的客服工单 | 32 单/周 | ≤ 5 单/周 | 客服系统标签统计 |
| 支付页加载时间（P95） | 4.2s | ≤ 1.5s | Lighthouse / RUM 监控 |

### 功能要求

- 支付按钮点击后，页面在 300ms 内给出反馈（loading 出现）
- 支付结果页在 500ms 内展示，明确显示成功/失败状态
- 支付失败时展示具体原因（余额不足/网络超时/银行拒绝）
- 每种失败类型提供对应的行动按钮（重试/更换方式/联系客服）
- 支付处理中添加加载状态，防止重复提交

## 4. 边缘情况（Edge Cases）

| 场景 | 预期行为 |
|------|----------|
| 用户网络中断（支付请求发出后断网） | 15 秒后超时，提示"网络异常，请检查网络后重试"，保留订单状态 30 分钟 |
| 用户点击支付后快速关闭页面 | 后端监听 webhook，支付成功后发送短信/推送通知用户 |
| 用户连续快速点击支付按钮 | 前端按钮立即禁用，直到收到响应或超时（防抖 500ms） |
| 支付渠道（微信/支付宝）临时不可用 | 自动尝试备用渠道，或提示"该支付方式暂不可用，请尝试其他方式" |
| 用户余额不足但点击支付 | 明确提示"余额不足，差额 ¥xx"，提供"更换支付方式"和"去充值"选项 |
| 同一订单用户同时开两个窗口支付 | 第二个支付请求返回"订单已支付"或"正在处理中" |
| 支付成功但回调延迟（用户看到超时） | 页面轮询订单状态，支付成功后自动跳转成功页，无需用户重试 |
| 用户在支付过程中杀掉 APP/关闭浏览器 | 下次打开时，在首页显示"您有一笔未完成的订单"悬浮提示 |
| 支付回调被第三方重复发送 | 幂等处理，同一支付单号多次回调只处理一次 |
| 用户支付后马上申请退款 | 进入退款流程，原路返回，退款状态实时通知 |
| 大额支付触发风控（> ¥5000） | 提示"需身份验证"，要求短信验证或人脸识别 |
| 跨境支付汇率波动 | 订单创建时锁定汇率 15 分钟，超时需重新下单 |

## 5. 验证（Validation）

### 自动化测试
- [ ] 单元测试：支付状态机所有状态转换覆盖
- [ ] 集成测试：模拟微信/支付宝回调成功、失败、超时三种情况
- [ ] 集成测试：确认地理编码查询返回 5 个样本地址的结果
- [ ] 回退测试：确认当 API 返回 500 错误时，手动表单会显示
- [ ] 负载测试：确认第 95 百分位响应时间在 300 毫秒以内
- [ ] E2E 测试：用户从购物车到支付成功的完整流程（≤ 30 秒）

### 手动测试清单
- [ ] 飞行模式下点击支付，验证错误提示和重试流程
- [ ] 手机系统时间调快 1 天再支付，验证签名过期处理
- [ ] 支付过程中切换 WiFi/4G，验证订单状态一致性
- [ ] 同时打开两个浏览器窗口支付同一订单

### 上线监控（发布前 3 天）
- 每小时检查支付成功率波动（超过 ±3% 立即告警）
- 检查错误日志中是否出现新的未处理异常类型
- 客服团队确认重复支付工单是否减少

### 发布成功标准（发布后 7 天评估）
- 支付完成率 ≥ 91% 且维持 7 天
- 无 P0/P1 级支付故障
- 客服重复支付投诉 ≤ 5 单/周

## 案例 3：笔记应用（MindRecall）

### 背景
开发者张明想要创建一款知识管理工具，使用 AI 辅助完成完整的产品发现流程。

### 阶段 1：机会发现

**数据来源：**
- Reddit：r/productivity、r/smallbusiness（约 5 万条数据）
- App Store 差评：Notion、Trello、Asana
- Twitter：#productivity #frustrated

**AI 分析任务：**
```
1. 爬取过去 6 个月的文本数据
2. 提取"试图做...但是..."等模式的句子
3. 情感分析标记情绪强度（1-5分）
4. 按主题聚类
```

**发现的高价值问题：**

| 排名 | 问题描述 | 提及次数 | 情绪强度 |
|------|---------|---------|---------|
| 1 | 知识分散在各个工具，查找困难 | 1,247 | 4.3/5 |
| 2 | 任务提醒太多导致麻木 | 892 | 4.1/5 |
| 3 | 项目管理工具太重 | 756 | 3.8/5 |

**关键洞察：**
> "我在 Notion 里存了 300 多页笔记，但找东西时根本不知道在哪。搜索也不好用，最后总是重新 Google。"

### 阶段 2：机会评估

**RICE 评分：**

| 机会 | R | I | C | E | 得分 |
|------|---|---|---|---|------|
| 知识分散 | 8 | 9 | 7 | 6 | **84** |
| 提醒麻木 | 7 | 8 | 6 | 5 | 67 |
| 工具太重 | 6 | 7 | 8 | 7 | 48 |

**市场规模：**
- TAM：知识管理工具市场 $1,200 亿
- SAM：PKM 细分市场 $80 亿
- SOM：第一年目标 $100 万 ARR

**差异化定位：**
> 不做"另一个笔记工具"，而是做"自动帮你整理和找到知识"的智能管家。

### 阶段 3：用户研究

**Persona A：研究员小李**
- 28 岁，市场研究员
- 每天阅读 20+ 篇文章
- 痛点：写报告时找不到之前看过的参考
- 目标：30 秒内找到任何之前看过的内容

**JTBD：**
> 当我需要用到之前的知识时，我想在 30 秒内找到它，而不必记得它存在哪里、用什么标题。

**用户场景：**
- 场景 1：紧急查找（客户 2 小时后要报告，需要找到之前看过的文章）
- 场景 2：内容复用（为不同客户做类似分析，需要关联历史内容）

### 阶段 4：解决方案设计

**Kano 分类：**

| 功能 | 分类 |
|------|------|
| 保存网页/文章 | 基本需求 |
| 自然语言搜索 | 期望需求 |
| AI 自动关联内容 | 兴奋需求 |
| 自动提取关键信息 | 兴奋需求 |

**MoSCoW 优先级：**

**Must Have：**
- 内容抓取和存储
- AI 自然语言搜索
- 基础标签系统

**Should Have：**
- 自动内容关联
- 浏览器插件
- 移动端 App

**MVP 定义（SMART）：**
- S：能保存网页/文章，支持自然语言搜索
- M：用户能在 30 秒内找到目标内容
- A：3 个月内可完成
- R：直接解决"知识查找难"痛点
- T：8 周内发布内测版

### 阶段 5：PRD 文档

**用户旅程地图：**

```
┌─────────────────────────────────────────────────────────────┐
│ 用户活动   │   保存内容    │   组织整理    │   搜索使用    │
├───────────┼───────────────┼───────────────┼───────────────┤
│ 用户任务   │ 浏览器插件保存│ 自动打标签    │ 自然语言搜索  │
│           │ 移动端转发    │ 手动添加标签  │ 筛选排序      │
│           │ 手动创建      │ 创建文件夹    │ 查看相关内容  │
├───────────┼───────────────┼───────────────┼───────────────┤
│ 功能点    │ 网页抓取      │ AI 标签推荐   │ 语义搜索引擎  │
│           │ 去重检测      │ 标签管理      │ 结果高亮      │
│           │ 格式解析      │ 文件夹管理    │ 自动关联展示  │
└───────────┴───────────────┴───────────────┴───────────────┘
```

**用户故事：**

**US-002：自然语言搜索**
> 作为研究员小李，我想用自然语言描述要找的内容，以便找到记不清标题的文章。

**AC：**
- Given 用户已有保存的内容
- When 用户在搜索框输入自然语言描述
- Then 系统返回最相关的 5 个结果
- And 结果按相关度排序
- And 每个结果显示内容片段和匹配原因

### 阶段 6：IntentSpec

**功能**：自然语言搜索

**目标：**
- 找到特定内容平均需要 4.5 分钟
- 70% 用户尝试 3-5 个关键词
- 35% 用户最终放弃搜索

**成果指标：**

| 指标 | 当前值 | 目标值 |
|------|--------|--------|
| 平均搜索时间 | 4.5 分钟 | ≤ 30 秒 |
| 搜索成功率 | 45% | ≥ 80% |
| 搜索到放弃率 | 35% | ≤ 10% |

**关键边缘情况：**

| 场景 | 预期行为 |
|------|----------|
| 搜索词过于模糊（如"那个文章"）| 提示"请提供更具体的信息" |
| 搜索服务暂时不可用 | 降级到标题匹配搜索 |
| 用户连续搜索 5 次无结果 | 主动提供人工帮助入口 |
| 内容为图片/PDF，无文字 | 返回文件名，提示暂不支持内容搜索 |

**验证：**
- 单元测试：语义匹配算法准确率 ≥ 85%
- 集成测试：模拟 1000 条内容的搜索响应时间
- E2E 测试：完整搜索流程 ≤ 3 秒

### 成果

8 周后，张明推出内测版"MindRecall"：
- 首批 200 个测试用户
- 搜索功能 NPS：+45
- 85% 用户表示"终于找到了一个能真正帮我找到知识的工具"

## 学习要点

1. **不同场景选择不同深度**
   - 公司级产品：完整 6 阶段
   - 功能优化：直接从 IntentSpec 开始
   - 个人创业：AI 辅助快速验证

2. **数据驱动决策**
   - 电商案例：用流失数据说话
   - 支付案例：用客服工单量化问题
   - 笔记案例：用爬取数据分析需求

3. **边缘情况是区分专业与业余的关键**
   - 支付案例的 12 种边缘情况展示了金融级产品的严谨
   - 笔记案例的降级方案体现了对用户体验的考虑

4. **可衡量的成果**
   - 每个案例都有明确的数字目标
   - 不是"提升体验"，而是"转化率从 X 到 Y"
