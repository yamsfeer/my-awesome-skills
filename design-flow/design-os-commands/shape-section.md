# 规划模块

你正在帮助用户定义产品某个模块的规格。这是一个对话式流程，用于确定功能范围、用户流程和界面需求——然后自动生成规格和示例数据。

## 第一步：检查前提条件

首先，验证 `/product/product-roadmap.md` 是否存在。如果不存在：

"目前尚未定义产品路线图。请先运行 `/product-roadmap` 定义产品模块，然后再回来规划各个模块。"

如果路线图不存在，停止于此。

## 第二步：确定目标模块

读取 `/product/product-roadmap.md` 获取可用模块列表。

如果只有一个模块，自动选中它。如果有多个模块，使用 AskUserQuestion 工具询问用户想要处理哪个模块：

"您想要为哪个模块定义规格？"

以选项形式展示可用模块。

## 第三步：收集初始输入

确定模块后，邀请用户分享初步想法：

"让我们定义 **[模块标题]** 的范围和需求。

您对这个模块应该包含哪些内容有什么想法吗？分享您对功能、用户流程或界面模式的任何构想。如果还没有想法，我们可以从问题开始。"

等待他们的回复。用户可能会提供原始想法，也可能要求直接进行问答。

## 第四步：提出澄清性问题

使用 AskUserQuestion 工具提出 4-6 个有针对性的问题来定义：

- **主要用户操作/任务** — 用户在此模块中能做什么？
- **需要展示的信息** — 需要显示哪些数据和内容？
- **关键用户流程** — 具体的交互步骤是什么？
- **界面模式** — 是否需要特定的交互、布局或组件？
- **范围边界** — 明确排除哪些内容？

示例问题（根据他们的输入和模块灵活调整）：
- "用户在此模块中可以执行哪些主要操作？"
- "主视图上需要展示哪些信息？"
- "带我了解主要用户流程——步骤是什么？"
- "您想使用哪些具体的界面模式？（例如：表格、卡片、弹窗）"
- "这个模块明确不包含哪些内容？"
- "是否需要多个视图？（例如：列表视图和详情视图）"

每次提一两个问题，保持对话感。专注于用户体验和界面需求——不讨论后端或数据库细节。

## 第五步：询问外壳配置

如果已为此项目创建了外壳设计（检查 `/src/shell/components/AppShell.tsx` 是否存在），询问用户关于外壳使用的问题：

"这个模块的界面设计应该显示在**应用外壳内**（带导航标题），还是作为**独立页面**（不带外壳）？

大多数模块使用应用外壳，但某些页面（如面向公众的视图、落地页或嵌入式组件）应该是独立的。"

使用 AskUserQuestion，选项为：
- "在应用外壳内" — 大多数应用内模块的默认选项
- "独立（无外壳）" — 用于公共页面、落地页或嵌入式组件

如果尚未创建外壳设计，跳过此问题，默认使用外壳。

## 第六步：自动推进——创建规格和示例数据

从澄清性问题中收集到足够信息后，**立即推进**，无需征求确认。依次完成以下所有步骤：

### 6a：创建规格文件

在 `product/sections/[模块id]/spec.md` 创建文件，使用以下精确格式：

```markdown
# [模块标题] 规格

## 概述
[2-3 句话概述此模块的功能]

## 用户流程
- [流程 1]
- [流程 2]
- [流程 3]
[添加所有讨论过的流程]

## 界面需求
- [需求 1]
- [需求 2]
- [需求 3]
[添加所有讨论过的需求]

## 配置
- shell: [true/false]
```

**重要说明：**
- 如果模块应在应用外壳内显示，设置 `shell: true`（这是默认值）
- 如果模块应作为不带外壳的独立页面显示，设置 `shell: false`
- 模块 id 是模块标题的 slug 版本（小写，空格替换为连字符）
- 不要添加未讨论的功能，也不要遗漏已讨论的功能

### 6b：生成示例数据和类型

写完规格后，立即为此模块运行完整的示例数据生成流程：

1. **检查全局数据结构** — 如果 `/product/data-shape/data-shape.md` 存在，读取它。使用实体名称和关系作为一致性指南。

2. **分析规格** — 确定用户流程隐含了哪些数据实体，每个实体需要哪些字段，以及可以执行哪些操作（这些将成为回调 props）。

3. **创建 `product/sections/[模块id]/data.json`**，包含：
   - 包含每个实体和关系通俗描述的 `_meta` 部分
   - 真实可信的示例数据（不使用"Lorem ipsum"或"测试 123"）
   - 主要实体的 5-10 条示例记录
   - 多样化内容：混合长短文本、不同状态
   - 边界案例：至少一个空数组，一段较长的描述
   - 字段名一致、TypeScript 友好的结构

   必需的 `_meta` 结构：
   ```json
   {
     "_meta": {
       "models": {
         "entityName": "用通俗语言描述该实体代表什么。"
       },
       "relationships": [
         "描述模型之间如何关联"
       ]
     }
   }
   ```

4. **创建 `product/sections/[模块id]/types.ts`**，包含：
   - 从示例数据推断的数据接口（字符串、数字、布尔值、数组、嵌套对象）
   - 基于规格的状态/枚举字段联合类型
   - 以 `[模块名称]Props` 命名的 Props 接口，数据作为 props，每个操作对应可选回调 props
   - 回调 props 的 JSDoc 注释
   - 接口名使用 PascalCase，属性名使用 camelCase

### 6c：告知用户

创建所有文件后，展示摘要：

"我已为 **[模块标题]** 创建了以下文件：

1. **规格** — `product/sections/[模块id]/spec.md`
2. **示例数据** — `product/sections/[模块id]/data.json`（[X] 条记录）
3. **TypeScript 类型** — `product/sections/[模块id]/types.ts`

规格快速摘要：

**概述：** [2-3 句话摘要]

**用户流程：** [简要列表]

**示例数据包含：** [实体和记录数的简短描述]

请随时查阅这些文件。如需调整规格或示例数据，请告知我。准备好后，运行 `/design-screen` 为此模块创建界面设计。"

## 重要说明

- 保持对话感和亲切感，不要机械化
- 当回答模糊时，提出追问
- 专注于用户体验和界面——不讨论后端、数据库或 API 细节
- 保持规格简洁——只包含讨论过的内容，不要膨胀
- 格式必须完全匹配，应用才能正确解析
- **不要**事先展示草稿征求确认——收集足够信息后直接写入文件
- 如果用户在审阅后提出修改，立即更新相关文件
